# 交易所架構優化階段與資料庫瓶頸分析

## 1. 垂直擴展 (Vertical Scaling) 的物理上限

當我們嘗試透過增加資料庫的 `max_connections` (最大連線數) 和 `shared_buffers` (共享緩衝區) 來提升效能時，最終會遇到以下物理限制：

- **磁碟 I/O 極限 (IOPS)**：即使關閉了 `synchronous_commit`，最終資料還是要寫入實體磁碟。在單機環境下，磁碟（即使是頂級 NVMe SSD）的隨機讀寫次數也是有硬性天花板的。
- **記憶體與 CPU 佔用**：每個資料庫連線都會消耗專屬的記憶體。當連線數過高（例如 > 500），作業系統核心進行 Context Switch（環境切換）所耗費的 CPU 資源，甚至會高過實際處理資料庫查詢的資源。
- **鎖競爭 (Lock Contention)**：在關聯式資料庫內，如果所有高頻請求都試圖同時修改同一個用戶的餘額（例如熱點錢包），資料庫必須進行 **Row-level Lock (行級鎖定)**。這會強迫併發請求變成循序排隊，這種機制下，堆疊再多硬體也無法讓同一個 Row 的並行寫入速度無限制增加。

## 2. 生產級交易所對「資料庫調教」的四大重點與具體作法

對於真正的金融級產品，單靠修改設定檔是不夠的，我們通常會採取以下進階架構手段：

### A. 引入連線池代理 (Connection Pooling)
- **概念**：不要讓微服務直接連線資料庫。PostgreSQL 每開啟一個連線都會 Fork 一個 Process，建立連線的成本非常高昂。
- **具體作法**：
  1. 在資料庫前方部署 **PgBouncer** 或 **Pgpool-II** 作為連線池閘道。
  2. 微服務（如 Order Service）統一將資料庫連線指向 PgBouncer，由 PgBouncer 在背後對資料庫維持少量且穩定的物理連線（例如 100-200 個）。
  3. 配置 PgBouncer 為 Transaction pooling 模式，這樣數千個來自微服務的短時間查詢都能安全地複用這 100 個物理連線，防止 DB 連線數爆發。

### B. 磁碟分級與 WAL 優化 (Disk Sharding & WAL Tuning)
- **概念**：將資料庫的隨機寫入（資料檔）與順序寫入（WAL 預寫日誌）物理隔離，減少磁頭尋軌延遲與 I/O 競爭。
- **具體作法**：
  1. 在雲端或實體機上準備兩顆獨立的高性能 NVMe SSD 磁碟。
  2. 將 PostgreSQL 的資料目錄 (`PGDATA`) 掛載至第一顆 SSD。
  3. 將 `pg_wal` (PostgreSQL Write-Ahead Logging 目錄) 透過 Symlink 或者掛載的方式移至第二顆 SSD。
  4. 這樣能確保系統在寫入日誌時專心做極速的順序寫入 (Sequential Write)，不會與資料頁的隨機寫入 (Random Write) 搶奪磁碟 I/O。

### C. 表分區 (Partitioning)
- **概念**：當 `orders` 或 `trades` 表累積到數千萬甚至數億筆時，單表查詢與寫入索引維護會變得異常緩慢。透過分區將大表拆分成多個小邏輯表。
- **具體作法**：
  1. 依照 `symbol` (交易對) 或 `created_at` (時間範圍) 進行 **Declarative Partitioning (宣告式分區)**。
  2. 例如建立分區表：`CREATE TABLE orders_btc_usd PARTITION OF orders FOR VALUES IN ('BTC-USD');`
  3. 未來當系統寫入或查詢 `BTC-USD` 的訂單時，資料庫只會掃描並鎖定該分區的數據區塊。不同交易對的操作（如 ETH-USD）將被完全隔離開來，大幅提升並發寫入與檢索效能。

### D. 讀寫分離 (Read/Write Splitting)
- **概念**：將繁重的讀取流量（查詢歷史紀錄）與核心的寫入流量（下單、扣款）強制分離，主庫只專心負責處理資料變更。
- **具體作法**：
  1. 設定 PostgreSQL 的 Streaming Replication，建立一台或多台非同步的 Read Replica 唯讀副本。
  2. 在應用程式層（或透過 ProxySQL/Pgpool）進行路由切分：
     - 所有的寫入操作（如 `POST /orders`）導向 Master 主節點。
     - 所有的查詢操作（如 `GET /orders/history`）全部導向 Replica 唯讀節點。
  3. 針對有強一致性需求的讀取，可設計「寫後讀 (Read-after-write)」策略，剛寫入的資料強制從主庫讀取，其餘則放寬至 Replica。

## 3. 架構天花板 (Ceiling of TPS)

這是一個技術選型的分水嶺，決定了系統擴展的最終極限：

- **傳統關聯式 DB 路徑 (PostgreSQL/MySQL)**：
  - **極限**：大約在 **10k ~ 30k TPS**。
  - **說明**：這是大多數中小型交易所及一般電商的物理極限。當你在這個階段卡關，單純調優資料庫參數已經無法突破，必須改變系統底層架構。

- **記憶體與事件溯源路徑 (In-Memory + Event Sourcing)**：
  - **極限**：百萬級別 **1,000,000 ~ 5,000,000 TPS**。
  - **說明**：這就是為什麼像 AXS 或 LMAX 這種頂級金融系統，**絕對不在核心下單路徑寫入關聯式資料庫**的原因。
  - **作法**：架構會先將請求送入**純記憶體狀態機（撮合引擎）**，並將結果事件（Event）非同步地依序寫入高效能 Msg Queue (如 Kafka) 或順序特化的日誌檔案（Append-only logs）。
  - **角色轉變**：在此架構下，關聯式資料庫退化成一個後端的「觀察者 (Consumer)」，專職從 Kafka 抓取結算結果做歷史備份與對帳用，它被完全移出極速撮合的核心關鍵路徑。

## 💡 總結與你的專案演進路徑

目前你的專案正處於從 **「第一階段 (DB-centric 關聯式架構)」** 向 **「第二階段 (Async-centric 非同步架構)」** 邁進的關鍵切換期：

- **你的優勢**：整體架構已經引入了 **Outbox Pattern** 和 **Kafka 撮合引擎解耦**，核心概念上完全具備了邁向百萬 TPS 「事件溯源架構」的骨幹。
- **目前的卡點**：為了確保資料交易（下單與錢包扣款）的 100% 絕對原子性安全，目前的 API 層仍採取「同步寫入 PostgreSQL」的策略，導致測試中受限於本地磁碟 I/O。

這展現了軟體工程中最真實的架構演進思考。在履歷或面試中，這是非常強大的論證武器：
> 「*我在高併發壓測中觀測到磁碟 I/O 成為單機瓶頸，因此我首先實踐了 Postgres WAL 調優與連線配置控制。為了突破關聯式資料庫的天花板，我設計了基於 Outbox Pattern 與非同步記憶體的撮合系統。這套架構不僅穩定解決分散式一致性難題，更保留了未來快速演進為完全 Event Sourcing 架構的延展彈性，具備挑戰 100k+ TPS 的潛能。*」
