# K6 高併發壓測與批次寫入優化實戰 (STAR 經驗紀錄)

這份文件記錄了在加密貨幣交易所後端專案中，我們從「能運作」邁向「能抗壓」的真實演進過程。在面對極限 K6 壓測時，我們遭遇了資料庫死結、連線池耗盡與系統假性瓶頸，並透過一系列架構與程式碼級別的調優成功突破。

這份紀錄非常適合作為面試時的 **STAR (Situation, Task, Action, Result)** 亮點實戰案例。

---

## 1. 情境與挑戰 (Situation & Task)

在我們完成核心的「Transactional Outbox + Kafka Event Sourcing」架構後，我們試圖使用 `k6` 進行 100 個虛擬用戶 (VU) 的極限負載測試 (Load Test)，目標是找出系統的 TPS 上限與首個崩潰點。

在壓測初期，我們遇到了以下嚴重的技術阻礙：

1. **資料庫連線池耗盡 (Connection Pool Exhaustion)**
   - **現象**：大量請求被 Nginx / Gateway 阻擋，後端狂噴 `context deadline exceeded` 或 `Gateway Context Canceled`。
   - **原因**：瞬間湧入的 HTTP 請求搶光了 PostgreSQL 的連線，等待取得連線的時間超過了 API 的 Timeout 設定。
2. **高併發下的資金鎖死結 (Deadlock on Fund Locking)**
   - **現象**：當大量買單/賣單試圖「同時」扣除同一個帳戶的 USDT 或 BTC 餘額時，Postgres 引發了 `deadlock detected` 錯誤。
   - **原因**：並行執行緒以不同的順序爭搶更新 `accounts` 表中的多條紀錄（例如 Thread A 先鎖 USDT 再鎖 BTC，Thread B 先鎖 BTC 再鎖 USDT），產生了循環等待 (Circular Wait)。
3. **K6 壓測「餘額不足」假性瓶頸**
   - **現象**：K6 壓測跑不到 3 分鐘就失敗（`http_req_failed > 10%`），錯誤原因是收到大量的 HTTP 500。
   - **原因**：壓測時用戶狂下單導致餘額耗盡，但當時 API 統一將底層的「餘額不足 (Insufficient Funds)」包裝為 `500 Server Error`，誤導了 K6 將其視為系統崩潰。

## 2. 優化行動 (Action)

為了解決上述問題，我們展開了「配置層、架構層、程式層」的三維度極限優化：

### Action 1. 資料庫連線池調校與抽象提取
- 將原本散落的資料庫連線字串，重構成統一的 `internal/infrastructure/db/config.go` (`DBConfig`)。
- 調優 pgxpool 參數，設置 `MaxOpenConns: 50-100` 以符合實體資料庫的核心數與連線承受力，並配置 `MinOpenConns: 5` 保持連線熱機，減少冷啟動的延遲。
- 統一了跨層的 `db.TxKey` Context 傳遞機制，確保在 Service 與 Repository 層切換時，不會發生幽靈連線或 Transaction 遺失。

### Action 2. 錯誤邊界重塑 (Error Handling Refactor)
- 導入 Go 1.13+ 的 `fmt.Errorf("%w")` 錯誤包裹機制，保留底層 Repository 拋出的 `domain.ErrInsufficientFunds`。
- 在 API Handler 層使用 `errors.Is(err, domain.ErrInsufficientFunds)` 進行「強型別比對」，將業務錯誤精準定址為 **HTTP 400 Bad Request**，隔離出真正的 HTTP 500。

### Action 3. 實作「批次寫入 (Ingestion Batching)」與「死結防禦」
針對造市商 (Market Maker) 這種高頻寫入客群，單筆 HTTP + 單筆 TX 的成本太高：
1. **開放 `/orders/batch` API**：允許單一個 HTTP Request 挾帶高達上百張訂單。
2. **`pgx.CopyFrom` 極速 I/O**：使用 PostgreSQL 的原生 Copy 協定取代百個獨立的 `INSERT` 語句，降低網路 Round-trip，展現出 10 倍以上的寫入效率。
3. **雙重記憶體排序 (防禦 Deadlock 殺招)**：
   - 在執行 `BatchLockFunds` 時，我們在 Go 記憶體階段，預先將所有要更新的帳戶餘額進行 **1) UserID 排序** 以及 **2) Currency 排序**。
   - 保證了無論外部請求多麼混亂，進入 Postgres 時的 `SELECT ... FOR UPDATE` 永遠依循固定順序，從根本上拔除了死結發生的可能性。

### Action 4. 重構 K6 架構級壓測體系
- 引進**「動態充值 (Dynamic Recharge)」**機制：當 K6 腳本收到我們優化過後的 HTTP 400 餘額不足錯誤時，腳本會自動呼叫後門 API 進行資金 Recharge 再重送，讓壓測能真正打滿全時段，榨出引擎物理極限。
- 梳理測試象限：定義了 `smoke-test` (業務巡檢)、`load-test` (大門抗壓)、`spike-test` (限流驗證)、`e2e-latency-test` (非同步延遲) 等等，各司其職。

## 3. 成果亮點與面試表達 (Result & Interview Highlights)

> 💡 這些成果可以直接放入履歷或在面試中作為「你做過最複雜的技術挑戰」來回答。

**亮點一：極致的非同步內部延遲 (Micro-Latency)**
"為了擴展系統，我設計了基於 Outbox Pattern 的非同步微服務架構。透過客製化 K6 腳本 (`e2e-latency-test.js`) 實測證明，即使面臨高強度資源競爭，訂單從寫入資料庫 $\rightarrow$ Outbox $\rightarrow$ Kafka $\rightarrow$ 撮合引擎 $\rightarrow$ Kafka $\rightarrow$ 推播，**整體 P95 核心事件流 (Event Stream) 延遲仍穩定控制在 30 毫秒以內**。"

**亮點二：百倍寫入極限 (Ingestion Batching & Deadlock Prevention)**
"在承接造市商流量時，我觀測到單次 HTTP 的 I/O 開銷成為瓶頸。因此我實作了批次寫入通道 (`BatchPlaceOrders`)，採用 Postgres `CopyFrom` 大幅降低網路往返。為了防止高併發下的資金死結，我在應用層實作了**二維資源排序演算法（依 UserID 與 Currency 排序）**，成功在高壓力的造市商批次腳本中展現了超越一般散戶入口十倍的寫入吞吐量，且達到 0 死結。"

**亮點三：高壓下的系統韌性 (Resilience)**
"在架構設計上，我們寧可拒絕用戶，也絕對不讓資料庫被塞爆。在 `spike-test` (突刺流量測試) 中，當 10 倍於負荷的流量衝入時，系統藉由 Gateway 限流與設定妥善的連線池策略，將超載部分優雅地以 **HTTP 429** 擋下，保證了系統 100% 不掉單且後端服務絕不發生 OOM 崩潰。"

## 4. 下一步：突破最終天花板
這次的壓測暴露出一個終極現實：**即便大門開得再寬（透過 Batching），傳統關聯式資料庫的 I/O 依然是撮合引擎的效能天花板**。
這為我們下一步將撮合引擎升級為 **「純記憶體撮合 (In-Memory Matching) 搭配盲寫 (Blind Write) 與事件隨從 (Event Follower)」** 提供了最堅實的數據驅動 (Data-Driven) 決策支持。
