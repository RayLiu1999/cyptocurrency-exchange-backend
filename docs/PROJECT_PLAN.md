# Go Exchange System - Side Project Plan

這份文件規劃了一個基於 Go 語言與 AWS 的分散式交易所 Side Project。目標是透過實作高併發、高可用的交易系統，在面試中展示分散式系統設計能力。

### 問題 3：分散式交易的資料不一致 (Consistency)

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  🔴 問題場景 (最嚴重的 Bug！)                                               │
├─────────────────────────────────────────────────────────────────────────────┤
│  用戶下單買 0.1 BTC，需要鎖定 $5000 USD                                     │
│                                                                             │
│  步驟 1: Wallet Service - LockFunds($5000) ──→ ✅ 成功                      │
│  步驟 2: Order Service - CreateOrder()     ──→ ❌ 失敗 (網路超時)           │
│                                                                             │
│  結果：用戶的 $5000 被鎖定，但訂單沒建立                                    │
│  用戶看到：錢不見了，訂單也沒有 → 客訴爆炸                                  │
└─────────────────────────────────────────────────────────────────────────────┘
                                      ↓
┌─────────────────────────────────────────────────────────────────────────────┐
│  🟢 解決方案：Saga Pattern + 補償機制                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│  Saga 流程：                                                                │
│  ┌──────────────────────────────────────────────────────────────────┐       │
│  │  1. Wallet.LockFunds()   ──→ 成功 ✅                              │       │
│  │  2. Order.CreateOrder()  ──→ 失敗 ❌                              │       │
│  │  3. Wallet.UnlockFunds() ←── 補償 (Compensate) 🔄                 │       │
│  └──────────────────────────────────────────────────────────────────┘       │
│                                                                             │
│  技術實作：                                                                 │
│  • Saga Orchestrator 協調器 (統一管理交易流程)                              │
│  • 冪等性設計 (Idempotency Key) - 重試不會重複扣款                          │
│  • 事件溯源 (Event Sourcing) - 可追蹤每筆交易的完整歷史                     │
│  • 最終一致性 (Eventual Consistency) - 允許短暫不一致，最終會一致           │
│                                                                             │
│  程式碼範例：                                                               │
│  ```go                                                                      │
│  func (s *Saga) PlaceOrder(order *Order) error {                            │
│      // Step 1: 鎖定資金                                                    │
│      if err := s.wallet.LockFunds(order.UserID, amount); err != nil {       │
│          return err                                                         │
│      }                                                                      │
│                                                                             │
│      // Step 2: 建立訂單                                                    │
│      if err := s.order.Create(order); err != nil {                          │
│          // 補償：解鎖資金                                                  │
│          s.wallet.UnlockFunds(order.UserID, amount) // Compensate!          │
│          return err                                                         │
│      }                                                                      │
│      return nil                                                             │
│  }                                                                          │
│  ```                                                                        │
└─────────────────────────────────────────────────────────────────────────────┘
                                      ↓
┌─────────────────────────────────────────────────────────────────────────────┐
│  🎤 面試故事                                                                │
├─────────────────────────────────────────────────────────────────────────────┤
│  "我實作了 Saga Pattern 處理分散式交易，                                    │
│   當訂單建立失敗時，自動觸發補償邏輯解鎖資金，                              │
│   確保用戶資金不會被錯誤鎖定，達到『最終一致性』。"                         │
│                                                                             │
│  進階回答：                                                                 │
│  "為了處理重試場景，我為每筆交易加入 Idempotency Key，                      │
│   即使網路重試導致重複請求，也不會重複扣款。"                               │
└─────────────────────────────────────────────────────────────────────────────┘
```

**關鍵程式碼位置**：
- `internal/core/service.go` - Saga 邏輯
- `internal/core/ports.go` - 介面定義

│  🔴 問題場景                                                                │
├─────────────────────────────────────────────────────────────────────────────┤
│  Elon Musk 發推："Bitcoin to the moon 🚀"                                   │
│  10 秒內湧入 100,000 筆下單請求                                             │

│  User → API → 直接寫入 DB                                                   │
│                                                                             │
│  結果：PostgreSQL 連線池耗盡 (max_connections: 100)                         │
│        寫入失敗，訂單丟失，用戶崩潰                                         │
└─────────────────────────────────────────────────────────────────────────────┘
                                      ↓
┌─────────────────────────────────────────────────────────────────────────────┐
│  🟢 解決方案：Kafka 訊息隊列 - 削峰填谷                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│  非同步處理架構：                                                           │
│                                                                             │
│  ┌──────┐    ┌─────┐    ┌───────────┐    ┌───────────┐    ┌────┐           │
│  │ User │───→│ API │───→│   Kafka   │───→│ Matching  │───→│ DB │           │
│  └──────┘    └──┬──┘    │   Queue   │    │  Engine   │    └────┘           │
│                 │       └───────────┘    └───────────┘                      │

│         立即回應：                                                          │

│                                                                             │
│  技術實作：                                                                 │
│  • Kafka 作為緩衝層 (Buffer)                                                │
│  • Consumer Group 平行消費 (多個 Worker 處理)                               │
│  • Back Pressure 背壓控制 (消費速度 < 生產速度時自動調節)                   │

│                                                                             │
│  流量對比：                                                                 │
│  ┌─────────────────────────────────────────────────────────────────┐        │
│  │  流量   ████████████████████████████████   (100k 請求)         │        │
│  │         │                                                       │        │
│  │  Kafka  ▼ 緩衝                                                  │        │
│  │  處理   ═══════════════════════════════════════   (穩定 1k/s)  │        │
│  └─────────────────────────────────────────────────────────────────┘        │
└─────────────────────────────────────────────────────────────────────────────┘
                                      ↓
┌─────────────────────────────────────────────────────────────────────────────┐
│  🎤 面試故事                                                                │
├─────────────────────────────────────────────────────────────────────────────┤
│  "我引入 Kafka 作為緩衝層，將同步下單改為非同步處理，                       │
│   成功應對 10 倍流量高峰，P99 延遲從 500ms 降到 50ms。                      │
│                                                                             │
│   當流量暴衝時，API 立即回應 'ACCEPTED'，訂單進入 Kafka 隊列，              │
│   後端 Worker 以穩定速度消費，保護資料庫不被打爆。"                         │
└─────────────────────────────────────────────────────────────────────────────┘

## 🎤 面試常見問題與標準回答

### Q1: "為什麼選擇微服務架構？"

```
回答框架 (STAR)：

Situation: "在單體架構下，撮合引擎的 bug 會導致整個交易所崩潰。"

Task: "需要實現服務隔離，讓單一服務故障不影響整體系統。"

Action: "我將系統拆分為訂單服務、撮合引擎、錢包服務三個獨立服務，
        使用 gRPC 進行服務間通訊，並引入 Circuit Breaker 防止連鎖故障。"

Result: "撮合引擎故障時，訂單服務仍可接收訂單存入 Kafka，
        實現了服務降級而非全站崩潰。"
```

### Q2: "如何處理高併發？"

```
回答框架：

"我採用了三個策略：

1. 水平擴展：API 服務無狀態設計，配合 K8s HPA 自動擴容，
   CPU > 70% 時從 2 pods 擴展到 10 pods。

2. 非同步處理：引入 Kafka 作為緩衝層，下單請求立即回應 'ACCEPTED'，
   後端非同步撮合，P99 延遲從 500ms 降到 50ms。

3. 快取優化：使用 Redis 快取熱門交易對的 OrderBook 快照，
   減少 DB 查詢次數，提升讀取效能 10 倍。"
```

### Q3: "如何保證資料一致性？"

```
回答框架：

"跨服務的分散式交易，我使用 Saga Pattern：

場景：下單需要『鎖定資金』+『建立訂單』兩個步驟

問題：如果鎖定成功但建立訂單失敗，資金會被錯誤鎖定

解決：
1. 每個操作都有對應的補償操作 (Compensate)
2. 訂單失敗時，自動觸發 UnlockFunds 補償
3. 使用 Idempotency Key 確保重試不會重複扣款

結果：達到『最終一致性』，即使部分失敗也能自動恢復。"
```

### Q4: "遇到過什麼技術挑戰？"

```
回答框架：

"最大的挑戰是『分散式交易的一致性問題』：

問題：錢包扣款成功，但訂單建立失敗（網路超時）

嘗試 1：Two-Phase Commit (2PC) → 效能太差，鎖定時間長

嘗試 2：Saga Pattern + 補償機制 → 成功！

關鍵學習：
- 分散式系統中，強一致性成本太高，『最終一致性』更實際
- 冪等性設計很重要，每個操作都要能安全重試
- 事件溯源讓我能追蹤每筆交易的完整歷史，方便 debug"
```

## 1. 專案目標 (Project Goals)

*   **核心功能**：實作一個具備下單、撮合、錢包管理功能的簡化交易所。
*   **效能目標**：
    *   撮合引擎延遲 < 10ms (99th percentile)。
    *   支援 1k+ TPS (Transactions Per Second)。
*   **架構目標**：從單體架構演進至微服務架構，最終部署於 AWS。
*   **面試目標**：產出可展示的架構圖、效能數據、以及解決分散式問題（一致性、併發）的具體案例。

## 2. 技術選型 (Technology Stack)

## 3. 階段性 Roadmap 與面試價值

### Phase 1: 單體 MVP (The Monolith)
**目標**：建立核心業務邏輯，確保正確性。
*   **功能**：REST API 下單 (Limit/Market Order)、記憶體內撮合引擎、資料持久化到 DB。
*   **架構**：單一 Go Process + PostgreSQL。
*   **解決的問題**：
    *   如何設計 Order Book 資料結構？(使用 Red-Black Tree 或 Skip List 優化查找)。
    *   如何處理浮點數精度問題？(使用 `decimal` 庫避免精度丟失)。
*   **面試價值**：展示對 Go 語言掌握度、資料結構與演算法能力。

### Phase 2: 微服務拆分 (Microservices Split)
**目標**：解耦系統，提升可維護性與擴展性。
*   **架構**：
    *   **API Gateway**: 統一入口，身分驗證 (JWT)。
    *   **Order Service**: 接收訂單，驗證，寫入 DB。
    *   **Matching Engine**: 純記憶體運算，專注撮合。
    *   **Wallet Service**: 管理餘額。
*   **通訊**：服務間使用 gRPC 呼叫。
*   **解決的問題**：
    *   單體架構的耦合導致開發困難。
    *   撮合引擎需要獨立擴展，不應被 I/O 綁死。
*   **面試價值**：展示微服務設計原則、API 定義 (Protobuf)、Docker 容器化能力。

### Phase 3: 非同步與事件驅動 (Async & Event-Driven)
**目標**：解決高併發下的效能瓶頸與資料一致性。
*   **架構變更**：引入 Kafka。
    *   下單後直接回傳 "Accepted"，非同步進入 Matching Engine。
    *   撮合成功後發送 "TradeExecuted" 事件，非同步更新 Wallet 和 Order DB。
*   **解決的問題**：
    *   **削峰填谷**：流量暴衝時，Kafka 緩衝請求，保護後端 DB。
    *   **分散式交易**：使用 Saga Pattern 或 TCC 處理跨服務 (訂單 vs 錢包) 的一致性。
*   **面試價值**：這是**含金量最高**的階段。展示對 Eventual Consistency、Message Queue、冪等性 (Idempotency) 設計的理解。

### Phase 4: 雲端部署與監控 (Cloud & DevOps)
**目標**：生產環境就緒 (Production Ready)。
*   **任務**：
    *   使用 Terraform 建立 AWS VPC, RDS, ElastiCache, ECS Cluster。
    *   設定 GitHub Actions 進行 CI/CD。
    *   架設 Grafana Dashboard 監控 TPS、Latency、Error Rate。
*   **解決的問題**：
    *   "Works on my machine" -> "Works on Cloud"。
    *   如何快速定位線上問題？(Distributed Tracing)。
*   **面試價值**：展示全端工程能力 (DevOps)，證明能獨立負責系統上線。

## 6. 關鍵術語解釋 (面試必備)

### 分散式系統術語

| 術語 | 英文 | 簡單解釋 | 在本專案中的應用 |
|------|------|----------|-----------------|
| 水平擴展 | Horizontal Scaling | 加機器而非升級機器 | API 服務無狀態，可複製多份 |
| 垂直擴展 | Vertical Scaling | 升級單機配置 | 撮合引擎使用更大記憶體 |
| 負載均衡 | Load Balancing | 分散請求到多台機器 | Nginx/ALB 分發流量 |
| 最終一致性 | Eventual Consistency | 不保證即時一致，但最終會一致 | Kafka + Saga Pattern |
| 強一致性 | Strong Consistency | 讀寫即時一致 | PostgreSQL ACID |
| CAP 定理 | CAP Theorem | 一致性、可用性、分區容錯只能三選二 | 選擇 AP (高可用 + 分區容錯) |
| 冪等性 | Idempotency | 重複執行結果相同 | Idempotency Key 防重複扣款 |
| 熔斷器 | Circuit Breaker | 服務故障時快速失敗 | 防止撮合引擎拖垮訂單服務 |
| 削峰填谷 | Peak Shaving | 用緩衝區平滑流量 | Kafka 緩衝暴衝流量 |
| 事件驅動 | Event-Driven | 用事件觸發動作 | TradeExecuted 事件更新錢包 |
| Saga Pattern | - | 長事務拆分 + 補償 | 下單失敗時解鎖資金 |

### Go 語言術語

| 術語 | 解釋 | 在本專案中的應用 |
|------|------|-----------------|
| Goroutine | 輕量級線程 | 每個 HTTP 請求一個 goroutine |
| Channel | goroutine 間通訊 | 撮合引擎的訂單佇列 |
| Interface | 行為抽象 | Repository 介面解耦 DB |
| Context | 請求生命週期管理 | 超時控制、取消傳播 |
| Defer | 延遲執行 | 確保 DB 連線關閉 |
